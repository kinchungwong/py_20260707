## What got built

Spikes (each with a headless `--selftest` + a preview PNG/plot; the audio ones also
have a real-device default run):

- `event_queue.py` — topology-A carrier: a lock-free, non-blocking `collections.deque`
  (NOT `queue.Queue` — its lock is an RT hazard). Verified no drop/reorder under
  flood, producer never blocks (~0.15 µs/push), transit latency ≤ one drain interval.
- `realtime_envelope_release.py` — **first sound.** Per-block ASR gate `Envelope` +
  phase-continuous `Voice`; click-free = ramp from the *current* level (release) /
  re-attack from current (retrigger).
- `polyphony_voices.py` — fixed 16-voice `PolySynth`; allocate = same-note / free /
  **steal the quietest**; `tanh` headroom (never clips). 16 voices ≈ 2.8 % of the
  block deadline. FFT confirms 3 notes → 3 independent peaks.
- `playable_instrument.py` — **first playable**: keyboard/mouse → `InputRouter` →
  `EventQueue` → `PolySynth` → sounddevice, ~30 lines of glue over four reused
  modules. Focus-loss = all-notes-off (never a cursor grab).
- `input_to_sound_latency.py` — measured the wired path. Device output latency
  dominates; the lever is **`latency='low'`** (~35 → ~9 ms), NOT a smaller blocksize.
- `piano_voice.py` — the batch `synth_note` timbre (inharmonic partials +
  per-partial decay + phase randomization) folded into a streaming `PianoVoice`,
  drop-in via `PolySynth`'s new `voice_factory`. Affordable (~26 % of the deadline
  at 16 voices). Held notes now decay like a piano, not an organ.

Refactor (backward-compatible): `Voice.note_on(freq)` / `note_off()` + a
`voice_factory` on `PolySynth` — made the voice **pluggable**; all prior selftests
stayed green. `playable_instrument` gained `--voice piano|sine` (default piano).

Policy: `../policy/locating_python_environment.md` — always run via `.venv/bin/python`;
packages are installed **by the human only** (user-edited). Linked from root `README.md`.

## Decisions & why (canonical facts are in `../memory/`)

- **Topology A holds** — the audio callback drains the queue directly; nothing forced
  the worker-thread topology B (still parked in `../speculations/`).
  → [[event-queue]]
- **`latency='low'` committed** for the OutputStream — the dominant latency lever
  (device output ~11 ms; requesting a small blocksize alone does nothing). Applied to
  `playable_instrument`. → [[latency]]
- **Voice pool = 16**, steal the quietest; the piano voice's steal adds a *small*
  artifact (crossfade-steal deferred). → [[polyphony-voice-pool]], [[streaming-piano-voice]]
- **Default voice = `PianoVoice`**, `--voice sine` to A/B. → [[streaming-piano-voice]]
- **Two new memory domains:** `concurrency/` and `architecture/` (the latter holds
  [[signal-chain]], the end-to-end pipeline hub that ties the others together).

## What went well / what to watch

- **Composability paid off.** `playable_instrument` was ~30 lines over unmodified
  modules; the `voice_factory` refactor made `PianoVoice` a genuine drop-in. This is
  the return on the earlier shared-`piano_keyboard` widget + enum-tagged `NoteEvent`.
- **Audio spikes verified device-free.** Driving the real callback with scripted
  events and asserting on the rendered buffer (envelope shape, click-free via bounded
  sample-step, FFT peaks) let every audio spike run headless in CI — only the true
  device-output latency needed real hardware.
- **The `perf_counter` stamp threaded end to end** (captured at router emit) is what
  made the latency spike a measurement, not a guess.
- **Watch:**
  - Promotion out of `spikes/` is overdue — the code is maintained-app-shaped now.
  - The demo `_play()` in `realtime_envelope_release` / `polyphony_voices` still open
    streams at the *default* latency (~35 ms); only `playable_instrument` uses
    `latency='low'`. Cosmetic, but inconsistent.
  - **Latency tail = where CPython bites:** occasional ~28 ms queue-latency spikes
    from GC/scheduling. Fine at the mean; watch under heavy polyphony/load.
  - `piano_voice`'s RNG is unseeded (phases differ per run); selftest asserts are
    phase-independent so it's stable, but note it if a test ever looks flaky.
