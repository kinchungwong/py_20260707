## Key decisions, and why

- **Chord mode + staged entry, as one hybrid design.** Combines "one key sounds a whole
  chord" (chord-mode presets) with "build a chord from keys pressed at different real
  times, fire it together" (staged entry) — because staged entry turned out to be a
  natural *authoring path* for chord-mode presets, not a separate feature.
- **`shift-click` = forget, everywhere.** One gesture, two targets (un-stage a key,
  unbind a preset) — chosen over separate gestures per target because it's one thing to
  learn, not two.
- **Audition on stage, resolved in favor of sound over silence.** A short beep
  (~0.1–0.2s) when a key is staged, so the player is reminded what it sounds like before
  committing.
- **Space bar → `Play chord`.** Not a new mode (the human caught this mid-thought) —
  just a keybinding for an existing control, chosen because it reaches for the
  near-universal DAW/transport convention (space = play).
- **`Release` reuses the synth's own vocabulary** — the envelope state machine already
  has a `RELEASE` stage; the button borrows the term rather than inventing "Clear."
- **Next step is a spike, not a plan directly.** Nothing in the design has touched real
  audio, `InputRouter`, or the actual shipped keyboard mapping — the human chose to
  prototype a minimal real version first, matching the project's established
  discussion → spike → plan path, rather than writing a `claude/plans/` document from
  the sketch alone.

## Handoff: the pre-implementation gap list

Before the next session turns this into a spike, these are open — grouped by what kind
of gap each is. None were resolved today; they're the explicit starting agenda.

**A. Wiring into the real architecture (the largest gap — nothing here has touched real
audio yet)**
- How `staged` / `presets` map onto the real `NoteEvent` / `EventQueue` / `InputRouter`
  pipeline and the `sounder_id` identity model — a staged-but-unplayed note isn't quite
  a note-on or a note-off in the existing vocabulary.
- Whether the audition beep is a distinct lightweight audio path or a real `PolySynth`
  voice with a forced-short envelope.
- Whether the sketch's simplified `A S D F G H J K` / `Z X C V` layout matches, or must
  be reconciled against, the real app's actual shipped keyboard mapping.
- How `MouseGlissando` interacts with staged mode — does a drag stage multiple notes, or
  is staging click-only?
- Whether staged/saved notes store as tuning-relative (scale degree) or absolute Hz —
  matters because Just Intonation already ships; does a tonic change re-resolve saved
  presets or leave them locked?

**B. Interaction gaps**
- Only `space → Play chord` has a keybinding; `Save chord`, `Release`, `Reset`, and mode
  entry itself are mouse-only — worth deciding given keyboard real estate is the scarce
  resource that started this whole design.
- OS key-repeat (a held key fires repeated keydown events) — does that spam-stage or
  spam-beep? Not designed.
- The quasimode commit-gesture candidate (modifier-held, release-fires) was never
  tried — the sketch only demonstrates persistent-toggle-plus-explicit-commit.
- Whether there's an instrument-appropriate "undo" beyond the forget gesture — still
  open from `audience-and-accessibility.md`.

**C. Product/process decisions**
- Do saved chord presets persist across app restarts, or are they session-only?
- The input policy mandates all-notes-off on focus loss — should that also clear staged
  notes and/or forget presets, or is lingering staged state acceptable?
- No onboarding/discoverability answer yet for *why* staged mode exists to a first-time
  player — the audience principle asks for more than correct mechanics.
- A terminology/UI-copy audit of the locked-in design strings (`Save chord → Z`,
  `Presets full`, "forget") against `audience-and-accessibility.md` and
  `terminology-policy.md` — reused-but-unglossed isn't the same as reused-and-explained.

**D. Process**
- **Decided:** prototype as a real spike before writing a plan. Not yet decided: which
  slice is the minimal spike — staging + audition + one preset slot was the working
  guess in-session, not locked.

## What went well

- **Iterating live against a real, clickable sketch** (the `mcp__visualize` widget)
  surfaced design questions text-only discussion wouldn't have — e.g. the empty-tray
  placeholder need, and the fact that "which commit gesture" got implicitly answered by
  *building*, not by *deciding*, which the design doc now flags explicitly rather than
  hiding.
- **Catching an implicit decision honestly.** The design doc calls out that it settled
  on persistent-toggle-plus-explicit-commit without that being a deliberate choice —
  worth preserving that kind of self-audit as a habit.
- **Real terminology grounding paid off twice** — once against the outside world
  (musician/synth vocabulary) and once against the project's *own* code (`RELEASE`
  already existing in the envelope state machine).
